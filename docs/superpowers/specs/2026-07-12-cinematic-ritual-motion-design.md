# 问爻电影级摇卦动效设计规格

- **日期：** 2026-07-12
- **状态：** 技术路线已由用户确认
- **适用范围：** 单轮摇卦仪式、水墨手掌、三枚乾隆通宝、定爻交互
- **运行环境：** Electron 43、React 19、React Three Fiber 9、Three.js 0.185、GSAP 3.15

## 1. 目标与质量定义

本次重制以“影院动画级完成度”为质量目标，不把品牌名称当作无法检查的形容词。完成度必须由以下可观察结果证明：

- 手掌动作具备预备、蓄力、释放、惯性和收势，不以两张静态图交叉淡化冒充开合动画。
- 三枚古钱具有真实方孔、正反面、边缘厚度、文字浮雕、磨损、铜锈、法线和粗糙度差异。
- 古钱运动遵循可信的重量、抛物线、角动量、碰撞、衰减反弹和落定节奏，同时始终收敛到程序预先生成的逻辑结果。
- 手掌、古钱、提示文字、音效、结果和确认按钮共用一条可跳过、可重播、可清理的主时间轴。
- 连续六轮中，即使相同位置的古钱连续得到相同正反面，也必须完整重新播放，不复用上一轮的动画终点。
- 画面点击“跳过”后，所有视觉对象立即进入同一个最终帧，不允许 UI 已显示结果而 3D 场景仍在运动。
- “定此爻”按钮在 disabled、hover、focus、pressed 和 enabled 状态下保持同一几何中心。
- 1920×1080 平衡档目标 60 FPS；不活动时停止连续渲染。

## 2. 品质边界

### 2.1 程序可以保证的部分

- 动画阶段、持续时间、同步、跳过和清理行为。
- 古钱真实几何、材质、高清度、镜头、灯光与确定性运动。
- 60 FPS 性能预算、减少动态模式、可访问性和重复六轮的稳定性。
- 手掌动画资产的播放、混合、降级、替换和与古钱的交接。

### 2.2 必须由美术资产保证的部分

- 手指在开合过程中的解剖正确性。
- 掌根、手腕和衣袖的连贯形变。
- 水墨边缘的飞白、渗化、墨量和逐帧一致性。
- 角色动画中的弧线、重心、蓄力和跟随动作。

当前仓库只有 `ritual-hands-closed.png` 和 `ritual-hands.png` 两张完整背景图，没有中间姿势、透明通道、手部骨骼或视频。因此工程实现可以先达到专业的运行时和古钱品质，但最终手掌表演不能在缺少逐帧资产的情况下被宣称为电影级完成。

## 3. 动作编排

### 3.1 第一爻完整仪式

总时长目标为 3.20 秒，使用 60 FPS 运行时；手绘资产以 24 FPS 或更高采样。

| 时间 | 阶段 | 手掌 | 古钱 | UI |
|---|---|---|---|---|
| 0.00–0.18s | 呼吸 | 合掌前轻微吸气和上移 | 隐藏于掌心 | 标题淡入 |
| 0.18–0.66s | 合拢 | 手腕先行、手指延迟跟随 | 掌内轻微晃动声 | 提示保持 |
| 0.66–0.98s | 蓄力 | 轻压、停顿、反向预备 | 初始姿态锁定 | 不显示结果 |
| 0.98–1.22s | 释放 | 掌心打开，手指形成由内向外的弧线 | 从掌心交接点出现 | 提示渐隐 |
| 1.22–2.22s | 飞行 | 手掌继续张开并产生衣袖惯性 | 三条不同贝塞尔弧线和不同角动量 | 只允许跳过 |
| 1.78–2.50s | 碰撞 | 手掌进入收势 | 依次接触、反弹、摇摆 | 碰撞时不提前揭示结果 |
| 2.50–2.82s | 落定 | 手掌停在开放姿势 | 最终面朝向相机，阻尼归零 | 显示字/背与爻名 |
| 2.82–3.20s | 确认 | 墨边轻微回流后静止 | 完全静止 | 固定位置启用“定此爻” |

三个接触时刻必须错开 60 至 140 毫秒，避免三枚古钱像复制对象同时落下。

### 3.2 第二至第六爻

总时长目标为 2.20 秒。保留预备、释放、飞行、碰撞、落定五个动作逻辑，只缩短停顿，不删除动作阶段：

- 0.00–0.46s：合拢与蓄力。
- 0.46–0.68s：释放。
- 0.68–1.56s：飞行。
- 1.28–1.92s：错峰碰撞和衰减反弹。
- 1.92–2.20s：结果揭示与确认按钮启用。

### 3.3 动画原则

- **Anticipation：** 抛币前有清晰反向预备，不能从静止直接弹开。
- **Arcs：** 手腕、指尖和古钱质心都沿弧线运动，禁止纯线性位移。
- **Overlap：** 手腕先动，手掌随后，指尖最后到位；三枚古钱错峰离手和碰撞。
- **Follow-through：** 古钱离手后衣袖和墨边仍继续运动 6 至 10 帧。
- **Ease：** 不使用单一 `ease-out` 覆盖全程，每个阶段采用独立曲线。
- **Settle：** 古钱最后 300 至 500 毫秒必须出现幅度递减的摇摆，不可突然冻结。
- **Silhouette：** 关键姿势在无纹理剪影下仍能清楚区分合掌、蓄力、释放和开放。

## 4. 水墨手掌资产管线

### 4.1 推荐制作方式

首选真人手部表演参考加逐帧水墨处理：

1. 以固定镜头拍摄合掌、摇币、开掌和收势动作，至少 60 FPS，保留手腕和衣袖。
2. 在 After Effects、Blender Grease Pencil 或同级动画工具中完成轮廓跟踪和关键姿势修正。
3. 先在纯黑剪影层验收动作弧线，再添加飞白、墨晕、烟雾和纸纹。
4. 手掌和背景分层输出；背景不得烘焙进每一帧。
5. 以 straight alpha 输出母版，避免半透明边缘出现黑边。

### 4.2 资产规格

- 母版画幅：2560×1440，16:9。
- 安全区域：中央 70%，避免窗口裁切丢失指尖。
- 母版帧率：24 FPS 或 30 FPS；运行时由浏览器在 60 Hz 合成。
- 完整片段：至少 77 帧（24 FPS × 3.20 秒）。
- 短片段：至少 53 帧（24 FPS × 2.20 秒）。
- 色彩空间：sRGB 交付，线性空间合成确认。
- 透明母版：16-bit RGBA PNG 序列或 ProRes 4444。
- Electron 主交付：与实时场景共用同一背景底板的不透明 VP9 WebM，以规避 Alpha 解码和边缘合成差异。
- 可选交付：带 Alpha 的 WebM，仅在目标 Windows 设备完成解码兼容测试后启用。
- 回退交付：交接首尾帧 PNG；当前两张静态图通过“墨幕完全遮挡后切镜”使用，禁止直接交叉淡化。
- 背景：制作母版中独立交付；不透明运行时视频与实时场景必须引用同一背景资产 ID。

### 4.3 关键姿势验收

至少保留以下 12 个可独立审阅的关键姿势：

1. 开始呼吸；2. 手腕内收；3. 指尖接触；4. 完全合掌；5. 蓄力压缩；6. 反向预备；7. 掌根分离；8. 指尖延迟张开；9. 古钱离手；10. 最大开放；11. 衣袖跟随；12. 收势静止。

任何自动生成或插帧结果只要出现以下一种情况即退回重制：手指数变化、手指粘连、左右手身份交换、掌心翻转、关节逆折、腕部漂移、背景呼吸、墨纹闪烁或古钱穿过手掌。

## 5. 乾隆通宝资产

### 5.1 几何

技术样片阶段使用 `THREE.Shape` 加 `Path` 方孔构造真实带孔截面，再由 `ExtrudeGeometry` 形成厚度和倒角。正式美术资产可以替换为 Blender 导出的 GLB，但必须保持相同局部坐标和面朝向约定。

- 外圆、内方孔均为真实拓扑，不用黑色方块贴图伪装。
- 正面、背面和边缘使用不同材质索引。
- 外缘、方孔内缘和币面都有倒角，以产生可信高光。
- 单枚运行时三角面目标 20,000 至 35,000；三枚共用一份 BufferGeometry。
- 直径、厚度和方孔比例固定在资产清单中，三枚只允许细微材质差异。

### 5.2 正反面

- 汉字面按“乾、隆、通、宝”上、下、右、左排列。
- 满文面使用经过人工核对的字形资产，不以乱码或通用 Unicode 字符冒充铸文。
- 文字采用浮雕高度、法线或位移表达，不只绘制平面文字。
- 最终正面约定必须写入 `CoinFace` 到四元数的唯一映射，并由测试锁定。

### 5.3 材质与清晰度

- 美术母版：4096×4096 Base Color、Normal、Roughness、AO、Metalness。
- 默认运行时：2048×2048 KTX2；高画质允许 4096×4096。
- 纹理使用 mipmap、正确色彩空间和设备允许的最大各向异性过滤。
- 铜色由物理材质和灯光共同产生，禁止高强度 emissive 制造橙色塑料感。
- 三枚古钱共享基础纹理，但使用 `visualSeed` 选择旋转后的污渍遮罩和轻微色差。

### 5.4 确定性轨迹

每枚古钱的视觉轨迹由 `(tossId, visualSeed, coinIndex, face)` 生成：

```ts
interface CoinTrajectory {
  launchPosition: [number, number, number];
  controlA: [number, number, number];
  controlB: [number, number, number];
  landingPosition: [number, number, number];
  spinAxis: [number, number, number];
  spinTurns: number;
  impactAt: number;
  bounceHeight: number;
  wobblePhase: number;
  finalQuaternion: [number, number, number, number];
}
```

同一 `visualSeed` 重播必须逐帧得到相同轨迹；不同轮次即使 `face` 相同，也必须因为 `tossId` 不同而重新创建动画上下文。

## 6. 运行时架构

### 6.1 阶段模型

```ts
type RitualPhase =
  | 'preparing'
  | 'hands-closing'
  | 'hands-releasing'
  | 'coins-flying'
  | 'coins-settling'
  | 'awaiting-confirmation';
```

`RitualScreen` 不再维护相互独立的 `opened` 和 `settled` 定时器。唯一的 `RitualTimelineController` 驱动阶段变化，所有异步事件携带当前 `tossId`，旧轮次回调必须被丢弃：

```ts
interface RitualTimelineController {
  play(): void;
  finish(): void;
  restart(): void;
  dispose(): void;
  getProgress(): number;
}
```

### 6.2 组件边界

- `RitualScreen.tsx`：编排语义 UI、键盘操作、确认权限和可访问文本。
- `useRitualTimeline.ts`：创建、重启、跳过和销毁 GSAP 主时间轴。
- `ritualMotion.ts`：持续时间、标签、缓动和阶段转换纯配置。
- `CoinScene.tsx`：Canvas、相机、灯光和生命周期，不保存业务 settled 状态。
- `QianlongCoin.tsx`：共享几何、分面材质和单枚古钱对象引用。
- `coinGeometry.ts`：真实方孔几何和材质分组。
- `coinTexture.ts`：高清纹理、法线、粗糙度和资源释放。
- `coinTrajectory.ts`：从视觉种子生成纯确定性轨迹。
- `InkHands.tsx`：透明视频、序列帧和静态降级的统一资产接口。
- `ritualAssets.ts`：资产清单、版本、尺寸、交接帧和回退策略。

### 6.3 GSAP 与 R3F 协作

R3F 通过 `CoinRigAdapter` 暴露三枚古钱的 `Object3D` 引用；GSAP 直接驱动 position 和 quaternion，并在更新时调用 `invalidate()`。删除当前基于 `elapsedTime` 的 `useFrame` 动画，禁止同时存在第二套时钟。

主时间轴必须包含标签：

```text
start
handsClosed
release
coinsAirborne
firstImpact
lastImpact
settled
reveal
confirmable
```

`finish()` 将时间轴定位到 `confirmable` 并执行所有必要回调。React 卸载或 `tossId` 变化时先 `kill()` 时间轴，再释放纹理、监听器和对象引用。

### 6.4 UI 稳定性

确认按钮由固定定位容器负责居中，hover/pressed 动画只作用于内部元素：

```text
.ritual-confirm-slot 负责 left: 50% 与 translateX(-50%)
.ritual-confirm-button 只允许 translateY、scale、颜色和阴影变化
```

这样 hover 不会覆盖居中 transform。按钮只有 `awaiting-confirmation` 阶段可点击，点击后立即进入 `confirming` 并锁定。会话层以 `expectedTossId` 原子完成“确认当前爻并生成下一轮或成卦”，重复提交同一 `tossId` 必须成为无操作，不能只依赖按钮 disabled 防双击。

## 7. 性能与降级

- 平衡档 DPR 上限 1.5，高档上限 2.0，低档固定 1.0。
- 动画阶段目标 Draw Calls 不超过 45，三角面不超过 150,000。
- JavaScript 帧耗时 P95 小于 3 ms，总帧耗时 P95 小于 16.7 ms。
- 主动画结束后 Canvas 使用按需渲染，不保留 `Float`、持续 Sparkles 或无意义时钟更新。
- 阴影贴图只在需要时更新；古钱落定后冻结阴影。
- `prefers-reduced-motion` 模式在 200 ms 内直接淡化到最终姿势，仍保留结果揭示和可访问文本。
- WebGL 失败时显示高清正反面静态币图，业务结果和确认流程不受影响。

## 8. 测试与验收

### 8.1 纯函数测试

- 相同输入生成完全相同的三条轨迹。
- 不同 `tossId` 且相同 `face` 仍生成不同轨迹。
- 三枚古钱落点不重叠，接触时刻错开 60 至 140 毫秒。
- `text` 与 `reverse` 的最终四元数分别落到唯一正确面。
- 轨迹在任意 `t ∈ [0,1]` 下不穿出相机裁剪面。

### 8.2 组件测试

- 新 `tossId` 必须销毁旧时间轴并创建新时间轴。
- 点击舞台调用 `finish()`，结果、古钱和按钮在同一帧进入最终状态。
- 动画完成前按钮 disabled；完成后 enabled；确认后立即再次 disabled。
- 连续两轮相同三枚 `face` 仍完整重播。
- 减少动态模式不等待完整片段。
- hover、focus 和 pressed 前后按钮中心点误差小于 0.5 CSS px。

### 8.3 视觉回归

- 在 0.25× 慢放下检查手指、手腕、古钱离手和三次碰撞。
- 截取 `release`、`firstImpact`、`lastImpact`、`settled` 四个固定时间点。
- 在 1280×720、1920×1080、2560×1440 三种窗口验证裁切和清晰度。
- 对全部八种三币组合至少各重播三次。
- 用户最终验收必须基于实际运行样片，不以静态设计稿或单元测试代替。

### 8.4 性能验收

- 连续完成六爻，不出现首次 Shader 编译停顿。
- 记录 60 秒帧时间；P95 达标且无超过 50 ms 的连续三帧。
- 落定后 2 秒内 CPU/GPU 进入接近空闲状态。
- 检查纹理和几何释放，六轮后显存不持续增长。

## 9. 交付顺序

1. 先完成纯确定性轨迹、真实方孔几何和统一时间轴，以灰模资产验证动作。
2. 再完成高清古钱程序材质和镜头灯光，产出可运行技术样片。
3. 接入透明手掌资产接口，用当前静态资源验证降级路径。
4. 制作或取得 12 个关键姿势和完整透明动画，替换临时手掌资产。
5. 完成慢放、视觉回归、性能和六轮压力验收。

第 1 至 3 步完成代表“运行时技术样片完成”；只有第 4、5 步及用户实际样片验收通过，才能称为电影级最终动效完成。
