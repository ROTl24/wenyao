# 六爻占卜 Windows 桌面应用设计规格

- **日期：** 2026-07-11
- **状态：** 分段设计已确认，等待正式文档审阅
- **首发平台：** Windows x64
- **目标用户：** 用户本人及少量内部人员

## 1. 产品目标

构建一个视觉表现突出、排盘计算可靠、古籍依据可追溯的六爻占卜桌面应用。用户输入所占之事后，通过六轮水墨手影与三枚乾隆古币的交互完成起卦；程序以确定性规则完成排盘，再从用户提供的约 10 本固定古籍中检索证据，由云端 AI 生成结构化解卦报告，并支持围绕同一卦象继续追问。

产品成功不以“模型看起来懂六爻”为标准，而以以下结果为标准：

- 六爻结果和排盘事实由代码产生，错误率为零。
- AI 的古籍判断可以回到真实书页、真实章节和真实原文。
- 动效具有鲜明的东方水墨气质，且在普通 Windows 集成显卡上流畅运行。
- AI、网络或显卡能力异常时，核心起卦和排盘仍可使用。
- 本地密钥、问题和解卦记录得到明确保护，不被写入普通日志。

## 2. 已确认约束

- 首版只支持 Windows x64，不同时开发 macOS 或 Linux 版本。
- 首版是内部工具，不包含账号、支付、订阅、运营后台和云同步。
- 古籍知识库由用户提供约 10 本固定资料，包含可复制文字文档与竖排繁体扫描 PDF。
- 古籍在发布前离线处理并随应用提供；首版用户不能在应用中上传新书。
- 古籍、排盘、历史记录和索引保存在本机；云端只承担 AI 推理与查询向量生成。
- 起卦结果由程序随机生成。动画展示结果，但不能决定或修改结果。
- 首版使用一套明确、版本化的传统纳甲六爻规则，不提供流派切换界面。
- AI 完成结构化报告后，用户可以围绕本次卦象继续追问。
- 视觉方向为“水墨玄境”，双手方向为“水墨手影”，整体体验采用“沉浸式分幕”。

## 3. 首版范围

### 3.1 包含功能

- 所占问题与事项类型输入。
- 自动记录北京时间与规则版本。
- 六轮数字铜钱起卦与逐爻确认。
- 本卦、变卦、动爻、纳甲、地支、六亲、世应、伏神、月建、日辰和旬空排盘。
- 完整排盘页面与本卦、变卦对照。
- 内置古籍关键词与语义混合检索。
- 带真实证据引用的结构化 AI 解卦报告。
- 基于同一卦象和证据链的后续追问。
- 原书页面查看、引用区域高亮和证据上下文展开。
- 本地历史记录搜索、恢复和删除。
- AI Provider、模型、API 地址和密钥设置。
- 音量、画质、减少动态和键盘操作设置。
- Windows 安装、卸载和版本升级能力。

### 3.2 明确不包含

- 用户注册、登录、多人协作和权限系统。
- 收费订阅、订单、支付和用量计费。
- 云端历史同步或跨设备同步。
- 用户自行上传、OCR、校订和管理古籍。
- 多流派排盘切换、自定义规则编辑器和专家批注后台。
- 完全本地大模型、随安装包分发数 GB 模型或 GPU 推理管理。
- 网页版、移动端和公开 SaaS 服务。
- AI 自动替代医学、法律、财务等专业意见的承诺。

## 4. 用户体验流程

### 4.1 问事

首页只保留一个明确任务：输入所占之事。字段包括：

- `question`：必填，10 至 500 个 Unicode 字符。
- `category`：必填，取值为 `career`、`wealth`、`relationship`、`health`、`study`、`lost_item`、`travel`、`other`。
- `cast_at`：由系统写入，使用 IANA 时区 `Asia/Shanghai`。

问题为空、少于 10 个字符或超过 500 个字符时，“开始起卦”保持禁用并显示明确校验原因。API 未配置不阻止起卦；应用只在进入 AI 分析时给出设置引导。用户通过校验并点击“开始起卦”后，系统立即创建持久化会话。

### 4.2 六轮起卦

完整状态由 XState v5 管理：

```text
boot.loading
  -> idle.editingQuestion
  -> casting.preparing
  -> casting.handsClosing
  -> casting.handsOpening
  -> casting.coinsFlying
  -> casting.coinsSettling
  -> casting.awaitingConfirmation
  -> casting.committingLine
       -> 未满六爻：casting.preparing
       -> 已满六爻：hexagram.assembling
  -> plate.ready
  -> analysis.requesting
  -> analysis.streaming
       -> analysis.complete
       -> analysis.error
```

每轮开始时，Rust 核心先通过操作系统安全随机源生成三枚铜钱结果并写入 SQLite；写入成功后前端才播放动画。动画使用独立的 `visual_seed` 决定轨迹、圈数、落点和粒子，最终朝向必须收敛到已保存的逻辑结果。

只有 `casting.awaitingConfirmation` 接受“定此爻”事件。双击、连点、动画未完成、窗口失焦和重复事件均不得新增爻。确认后从初爻开始自下而上生成爻线，并自动进入下一轮准备状态。

### 4.3 排盘与分析

第六爻确认后，六条爻线在中央组成完整卦象，再转入阅读工作台：

- 左侧：本卦、变卦、六爻专业排盘和关键关系。
- 右侧：AI 报告、流式进度、错误与重试。
- 底部或抽屉：古籍证据卡片、原文上下文和原书页查看器。
- 顶部：所占之事、起卦时间、规则版本、保存与返回历史。

AI 报告完成后显示追问输入框。追问始终绑定当前 `session_id` 和只读 `plate_id`，不能重新起卦或修改排盘。

### 4.4 历史记录

历史记录按起卦时间倒序展示，支持按问题文本、事项类型、本卦和变卦搜索。打开未完成会话时从最后一个已持久化状态恢复；打开已完成会话时直接进入排盘与报告页。删除操作需要二次确认，并级联删除该会话的投币、排盘、分析、对话和引用缓存。

## 5. 视觉与动效设计

### 5.1 美术方向

整体视觉使用“水墨玄境”：宣纸微纹、墨晕、山雾、旧铜、朱砂和少量暖灰。禁止使用高饱和仙侠蓝紫光、持续故障闪烁、过量金色描边和廉价粒子瀑布。

水墨手影保留真实手部动作和衣料惯性，但将皮肤与轮廓处理为墨影、烟雾和半透明边缘，避免写实手部的恐怖谷问题。长篇排盘、古籍原文和 AI 回复全部使用 DOM 文本，不绘制进 WebGL Canvas，以保证中文清晰、可复制、可搜索和可访问。

### 5.2 混合渲染

- 双手合拢、张开和释放古币使用预渲染不透明视频，背景与实时场景使用同一水墨底图。
- 首次完整手部片段约 2.5 至 3 秒；后五轮使用约 1 至 1.4 秒的缩短片段。
- 实时 Three.js 场景在视频下方预热，视频结尾和 3D 起始帧共用镜头、灯光和古币位姿。
- 通过 6 至 8 帧交叉淡化从手部视频切换到实时古币。
- 资源交付必须包含 `handoff.json`，记录相机 FOV、相机位姿、三枚币的位置与四元数、灯光、色调映射和交接时间点。

### 5.3 实时古币

- 三枚古币共用几何与纹理资源，但保留三个独立对象。
- 单枚低模控制在 20,000 至 30,000 三角面；字形、磨损、法线、粗糙度和 AO 主要烘焙到 2K KTX2 纹理。
- 首版不引入真实物理引擎。每枚古币使用确定性三次贝塞尔轨迹、6 至 12 圈旋转、2 至 3 次衰减反弹和 0.3 至 0.5 秒阻尼摇摆。
- 视觉正反面由逻辑结果计算出的最终四元数决定，不以碰撞模拟结果为准。
- 接触时触发少量铜屑、墨点、桌面光圈和按强度区分的碰撞音效。

### 5.4 节奏

- 第一爻：3.2 至 3.8 秒，用于建立仪式。
- 第二至第六爻：每轮 1.8 至 2.4 秒。
- 每次确认后立即显示爻名、铜钱组合和自下而上生长的爻线。
- 第六爻完成后，铜钱退向两侧，爻线放大并拆分为本卦、变卦和排盘信息。
- 进入阅读工作台后，3D 场景切换为按需渲染并保持静止，避免持续占用 GPU。

### 5.5 音频与可访问性

音频包含环境底噪、衣料轻擦、至少三组摇币变体、轻中重铜钱碰撞、定爻声、成卦声和 AI 报告完成提示。环境音、效果音和总音量可独立调整，完全静音不影响信息理解。

应用读取 `prefers-reduced-motion` 并提供显式“减少动态效果”开关。减少动态模式用 150 至 250 毫秒淡化代替手部、飞行和镜头推进。Enter/Space 可以开始与确认，所有关键操作有清晰焦点样式，音效和颜色不得成为识别卦象的唯一方式。

## 6. 技术架构

### 6.1 技术栈

- 桌面壳：Tauri 2，Windows 使用 WebView2 Evergreen。
- 前端：React、TypeScript、Vite。
- 实时视觉：React Three Fiber、Three.js `WebGLRenderer`、GSAP、react-postprocessing。
- 交互状态：XState v5。
- 核心与本地服务：Rust。
- 本地数据库：SQLite，启用 WAL、外键、事务迁移和 FTS5。
- 古籍离线处理：Python 3.12、PDFium/PyMuPDF、PaddleOCR、OpenCC 和自定义竖排解析器。
- 云端 AI：Rust 侧 Provider 接口，首版实现 OpenAI-compatible Chat、Embedding 和流式响应适配器。
- Windows 密钥：Credential Manager；应用日志使用结构化、可脱敏记录。

首版生产渲染基线是 WebGL2。WebGPU 只做能力探测和未来扩展，不成为首版验收依赖。

## 7. 确定性六爻核心

### 7.1 投币约定

首版内置且不可在界面更改的约定 ID 为 `qianlong_text2_reverse3_v1`：

- 乾隆通宝汉字面为 `text`，计 2。
- 背面为 `reverse`，计 3。
- 三枚之和为 6：老阴，阴爻，动。
- 三枚之和为 7：少阳，阳爻，静。
- 三枚之和为 8：少阴，阴爻，静。
- 三枚之和为 9：老阳，阳爻，动。
- 变卦只翻转老阴和老阳对应的动爻。

界面必须在设置说明中展示这一约定，避免用户把不同铜钱计数法混入同一会话。

### 7.2 规则包

首版规则包 ID 为 `wenwang_najia_v1`，采用传统文王卦纳甲六爻框架。规则包以显式静态表和纯函数实现，至少包含八卦阴阳、六十四卦映射、纳甲地支、五行、六亲、世应、伏神、六冲六合与动变计算。AI 文本、古籍 OCR 和用户问题不得进入规则计算。

历法规则 ID 为 `beijing_jieqi_zichu_v1`：

- 时区固定为 `Asia/Shanghai`。
- 月建依据节气边界，不以农历初一作为月界。
- 日界采用子初换日，即本地时间 23:00 进入下一干支日。
- 所有计算记录规则 ID、引擎版本、时区和原始时间戳。

这些规则在首版不提供切换界面，但数据结构保留 `rule_pack_id`、`calendar_rule_id` 和 `coin_convention_id`，保证未来新增流派时不会破坏旧记录。

### 7.3 排盘输出

核心返回只读 `DivinationPlate`：

```text
session_id
engine_version
coin_convention_id
rule_pack_id
calendar_rule_id
timezone
cast_at
raw_tosses[6]
base_hexagram
changed_hexagram
moving_lines
trigrams
line_details[6]
month_branch
day_stem_branch
void_branches
computed_facts
```

每个 `line_detail` 包含自下而上的爻位、6/7/8/9 数值、阴阳、动静、纳甲、地支、五行、六亲、世应和伏神。Plate 创建后不可更新；需要纠正时必须废弃原会话并重新起卦。

## 8. 古籍证据包

### 8.1 数据层次

古籍不是单一文本文件，而是五层可追溯数据：

1. 原始证据：源文件、版本、书名、来源和哈希。
2. 页面层：PDF 页码、原书叶码、页面预览图和校验值。
3. OCR 层：文字、坐标、列号、阅读顺序、置信度和 OCR 模型版本。
4. 校订层：原始 OCR、人工校订文本、现代标点和搜索别名，修改保留版本。
5. 语义检索层：规则、卦例、注解、术语、父子块、元数据、关键词索引和向量。

核心实体：

```text
Book(id, title, edition, source, file_hash)
Page(id, book_id, pdf_page, printed_page, image_path, checksum)
TextSpan(id, page_id, bbox, column_id, reading_order,
         ocr_raw, corrected_text, confidence, model_version)
Chunk(id, type, original_text, punctuated_text, search_alias,
      section_path, parent_id, school, tags, review_status)
ChunkEvidence(chunk_id, text_span_id)
Embedding(chunk_id, model_id, dimensions, vector)
```

引用最终必须落到书名、版本、卷章、PDF 页、原书叶码和页面坐标。应用展示书页预览图并根据 `bbox` 绘制高亮，不让模型生成页码。

### 8.2 离线构建

```text
资料登记与哈希
-> PDF 页面分类
-> 页面渲染、双页拆分、方向与形变校正
-> 文本直接提取或 PaddleOCR
-> 竖排阅读顺序重建
-> 低置信文字复核
-> 规则、卦例、注解、术语语义切分
-> 六爻元数据标注
-> FTS5 与向量构建
-> 完整性与引用回链校验
-> 只读证据包发布
```

原文、校订文和搜索归一化文本始终分离。OpenCC 只生成简繁与异体搜索别名，不能改写引用原文。语义块以完整规则和完整卦例为边界；子块用于精确检索，命中后补充父块与相邻上下文。

首批人工金标准至少覆盖 100 页，包含清晰、污损、繁体、竖排、夹注和生僻字页面。关键术语识别错误单独统计，不能被全文平均准确率掩盖。

## 9. 检索与 AI 生成

### 9.1 三路检索

排盘和问题被转换为三个检索意图：

- `rule`：用神、旺衰、生克、月日、动变和规则条件。
- `case`：相似占类、卦象、动爻和已验证卦例。
- `original`：卦名、术语、经典原句、繁简和异体精确匹配。

每路执行元数据过滤、FTS5 BM25、语义向量检索、RRF 融合、去重和父块补全。融合后的前 30 条候选交给云端模型进行受约束证据重排；模型只能返回候选证据 ID 和相关性分数。最终证据包最多包含 12 个语义块，并同时保留不同古籍之间的冲突证据。

固定语料的向量在离线构建时生成并写入证据包；运行时只为用户问题生成查询向量。若查询向量服务失败，系统退化到 FTS5 检索，并在报告页显示检索能力降级。

### 9.2 报告生成

云端模型接收：

- 用户问题与事项类型。
- 只读 `DivinationPlate`。
- 最多 12 个带 `evidence_id` 的证据块。
- 固定输出 JSON Schema 与禁止事项。

输出结构：

```text
plate_summary
yongshen_analysis
strength_and_relations
moving_line_analysis
classical_claims[]
synthesis
uncertainties[]
actionable_guidance[]
```

每条 `classical_claim` 包含 `text`、`evidence_ids` 和 `confidence`。后端拒绝模型生成的书名与页码，只使用证据 ID 从本地数据库渲染来源。

生成后执行两类校验：

- 排盘事实校验：模型提到的卦名、动爻、六亲、世应、月日和旬空必须与 Plate 一致。
- 证据校验：每条古籍判断必须存在证据，引用文字必须位于对应 TextSpan，证据必须能够支持该结论。

事实不一致时自动重新生成一次；再次失败则阻止展示错误报告，保留排盘并显示可重试错误。证据不足时固定使用“当前内置资料不足以支持该判断”，不得补写无来源结论。

### 9.3 追问

追问沿用同一 `session_id`、`plate_id` 和已生成报告。系统根据新问题决定复用现有证据或重新执行三路检索。上下文发送当前报告、最近 12 条对话和必要证据；更早消息保存在本地但不自动发送。每条回答继续执行排盘事实与证据 ID 校验。

## 10. 本地数据与安全

运行数据和只读语料分为两个数据库：

- `app.sqlite`：可写，保存会话、投币、Plate、分析、对话、引用和设置。
- `corpus.sqlite`：只读，随证据包发布，保存书目、页面、文本、语义块和索引。

应用数据库至少包含：

```text
sessions
tosses
plates
analyses
chat_messages
analysis_evidence
app_settings
schema_migrations
```

SQLite 启用外键、WAL 和事务迁移。每次模式升级前创建一次可恢复备份；迁移失败时继续使用旧版本并停止写入，不能半迁移启动。

AI Provider 设置包含 `base_url`、`chat_model`、`embedding_model` 和 `api_key_ref`。API 密钥只进入 Windows Credential Manager，SQLite 仅保存引用名。Rust 侧发起网络请求，前端 JavaScript 不接触原始密钥。

`base_url` 必须使用 HTTPS；只有显式指向 `localhost` 或 `127.0.0.1` 的开发配置允许 HTTP。`app.sqlite` 位于当前 Windows 用户的应用数据目录并继承用户目录访问控制。首版不宣称本地历史数据库具备独立于 Windows 账户的二次加密保护。

日志默认只记录时间、版本、模块、错误类型、会话随机 ID 和耗时，不记录用户问题、解卦正文、完整证据原文和密钥。用户删除会话时，关联投币、排盘、报告、追问和证据缓存在同一事务中删除。

古籍视为不可信输入。书中出现的指令、提示词或“忽略系统规则”等文本只能作为引用内容，不能改变系统消息、工具权限和输出约束。

## 11. 错误处理与恢复

- API 未配置：允许起卦和排盘；进入分析时打开设置引导。
- 网络不可用：保留排盘，显示离线状态，允许稍后重试。
- AI 流中断：保存已完成状态但不把半截内容标记为正式报告。
- 查询向量失败：使用 FTS5 降级检索并明确提示。
- AI 事实校验失败：自动重试一次；再次失败则阻止错误报告。
- 证据包缺失或哈希失败：允许起卦与排盘，禁止 AI 分析并显示修复指引。
- WebGL2 不可用：切换静态铜钱图与 DOM 按钮，逻辑结果和完整流程保持可用。
- 性能不足：依次降低 DPR、粒子、Bloom、动态阴影、景深和手部视频。
- 起卦中断：从数据库中最后一个已保存状态恢复，不重新随机。
- 数据库迁移失败：恢复备份，停止写入并给出错误日志位置。

错误信息使用可操作文案，必须说明发生了什么、哪些数据仍然安全以及用户下一步可以做什么，不使用笼统的“内部服务错误”。

## 12. 性能预算

平衡档目标设备为 Intel Iris Xe 或同等级 Ryzen 集显，窗口分辨率为 1920×1080：

- 动画阶段目标 60 FPS，单帧 16.7 毫秒。
- JavaScript 每帧不超过 3 毫秒，GPU 不超过 10 至 11 毫秒。
- 常态 Draw Calls 不超过 60，峰值不超过 80。
- 常态三角面不超过 300,000。
- 平衡档活动粒子 2,000 至 3,000，高档最多 8,000，低档不超过 800。
- 核心场景纹理显存不超过 128 MB。
- GLB 与纹理磁盘资源不超过 30 MB。
- 两段主要手部视频合计不超过 15 MB。
- 本地 SSD 首次可交互时间不超过 2.5 秒。
- 非动画状态使用按需渲染，使 CPU/GPU 接近空闲。

4K 窗口限制 3D 内部最大渲染像素约为 2560×1440，DOM 文字保持原生清晰。模型、纹理、音频预加载，关键纹理预上传，Shader 在首次仪式前预编译。

## 13. 测试与验收

### 13.1 六爻核心

- 穷举 `4^6 = 4096` 种六爻数值序列。
- 验证本卦、动爻、变卦以及“只翻动爻”不变量。
- 为六十四卦、纳甲、世应、伏神、月建、日辰和旬空建立黄金用例。
- 覆盖 22:59、23:00、节气交界和不同时区输入的历法边界。
- 排盘事实错误率必须为零。

### 13.2 起卦状态机

- 双击、连续点击、动画中点击和窗口失焦不会多确认爻。
- 六次确认严格生成六爻，自下而上顺序不变。
- 进程在每个状态被终止后均可恢复，不重新随机。
- 动画重播同时使用已保存的逻辑结果和 `visual_seed`，保证同一轮重播结果与轨迹一致。

### 13.3 古籍与检索

- 至少 100 页人工金标准，统计全文 CER、关键术语 CER、阅读顺序和坐标覆盖率。
- 至少 150 个带正确证据的问题，覆盖原句、现代同义问法、规则、卦例、冲突来源和无答案问题。
- 记录 Recall@5、Recall@10、MRR、nDCG@10 和无答案拒答率。
- 每个引用都可以回到正确书页和高亮坐标。

### 13.4 AI

- 排盘事实一致率 100%。
- 古籍判断引用覆盖率 100%。
- 无证据问题能够拒答，不伪造书名、章节和页码。
- 冲突古籍并列展示，不被模型静默合并。
- 至少两名懂六爻的评审者抽查依据充分性、流派一致性、推断合理性和不确定性表达。

### 13.5 视觉与桌面

- 1080p 平衡档在目标集显上达到 60 FPS，低档稳定 30 FPS。
- 首次投币无明显 Shader 编译卡顿。
- 视频与实时 3D 交接无明显位移、亮度和色差跳变。
- Windows 10 和 Windows 11 完成安装、启动、卸载和升级回归。
- 静音、减少动态、纯键盘和 WebGL 静态回退均能完成完整流程。
- AI 断网不影响起卦、排盘、历史和古籍查看。
- API 密钥不出现在数据库、日志、前端状态和安装包中。

## 14. 发布方式

首版输出 Windows x64 NSIS 用户级安装程序，使用 WebView2 Evergreen。内部测试版允许通过版本化安装包直接分发，并明确告知未签名构建可能触发 SmartScreen。稳定分发版使用 Authenticode 签名；Tauri 更新包使用独立更新签名密钥，并为该密钥建立离线备份。

程序升级不能覆盖 `app.sqlite`、用户设置和历史记录。只读语料通过 `corpus_version` 独立升级；新程序启动时验证应用版本、数据库模式和语料版本的兼容关系。

## 15. 开发顺序与风险门

实施必须先验证三个高风险点，再进入完整页面开发：

1. 六爻核心原型：完成投币映射、六十四卦和动变，并通过 4096 组合测试。
2. 古籍实验：选取 20 至 50 页真实资料，对可复制 PDF、竖排繁体扫描和关键术语进行 OCR 与切分验证。
3. 视觉原型：完成单轮水墨手影交接、三枚实时古币和集显性能测量。

三项验证通过后再实现完整状态机、排盘工作台、证据包、AI 报告、追问、历史和安装更新。这样可以在投入大量视觉与页面开发前确认排盘、资料和渲染三条主链均可落地。

本规格是系统级总设计，实施时拆成四份按顺序执行、分别验收的计划，避免把独立高风险子系统塞进一个超大计划：

1. 桌面基础与确定性六爻核心：Tauri 工程、SQLite、规则包、4096 组合测试、会话与静态起卦闭环。
2. 古籍证据包与检索：离线 OCR/校订工具、语料模型、FTS5、向量、证据回链和原页查看器。
3. AI 报告与追问：Provider、证据重排、结构化报告、事实/引用校验、流式交互和失败恢复。
4. 水墨视觉与 Windows 发布：手影、实时古币、音频、画质降级、性能验收、安装、签名和更新。

每份计划都必须产生可以独立运行和测试的增量；后一份计划只依赖前一份已经验收的接口和数据结构。

## 16. 最终验收定义

首版完成必须同时满足：

- 用户能从输入问题开始，连续完成六轮铜钱起卦并得到正确排盘。
- 任意一轮中断都可以恢复，且铜钱结果不发生变化。
- AI 能基于内置古籍生成结构化报告，并从每条观点跳转到真实原书页。
- 用户能围绕同一卦象继续追问，所有回答保持排盘事实一致。
- 网络、AI 或 WebGL 故障不会导致已确认的爻和排盘丢失。
- 水墨仪式在目标 Windows 集显上达到性能预算，低动态和键盘模式可完整使用。
- 安装程序可以在干净 Windows 环境完成安装、升级与卸载，密钥不被泄漏。

只有上述条件全部通过，才能把首版标记为可交付。
